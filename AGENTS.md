# Codex_Test 向けエージェントガイドライン

## リポジトリ概要
- このリポジトリは現在、Go プロジェクトのスケルトンです。追跡されているファイルは以下の通りです。
  - 高レベルのプロジェクトドキュメントを記載するためのプレースホルダーである `README.md`。
  - GitHub の Go テンプレートに従い、一般的な Go のビルド生成物を除外する `.gitignore`。
- 今後アプリケーションコードを追加する際は、慣例的な Go のディレクトリレイアウト（例: エントリーポイント用の `cmd/<app>`、共有ライブラリ用の `internal/` または `pkg/`）に従い、プロジェクトが拡張されても構造が予測しやすいようにしてください。

## 開発ワークフロー
- コードを追加したら速やかに Go Modules（`go.mod`）で依存関係を管理してください。モジュールは特別な理由がない限りリポジトリ直下で初期化します。
- Go のソースコードはコミット前に必ず `gofmt`（または `goimports`）でフォーマットします。
- 関数は小さく保ち、明示的なエラーハンドリングを推奨します。ライブラリコードに `panic` を導入するのは避け、代わりにエラーを返してください。
- 新しいパッケージを追加する際はユニットテストも併せて追加し、変更を送る前に `go test ./...` が成功することを確認してください。
- 新機能を導入して文脈が分かりにくくなる場合は、`README.md` を更新するか、パッケージレベルのドキュメント（`doc.go`）を追加してください。

## ツールと CI の期待値
- `.gitignore` テンプレートが想定するデフォルトに合わせ、Go 1.21 以上を優先して使用します。`go.mod` を作成したら、必要な Go バージョンをファイル内に明記してください。
- 追加のツール（リンター、フォーマッター、CI ワークフローなど）を導入する場合は、他の貢献者がローカルで再現できるように、ここに実行コマンドを記載してください。

## プルリクエストに関する注意
- 各変更の目的を PR 説明に記載し、実行したコマンド（例: `go test ./...`）を列挙してください。
- コミットは関連するファイル群ごとにまとめ、必要がない限りリファクタリングと機能追加を混在させないでください。
